## Estado del arte que hace el autor(1 página)

El autor menciona que en los últimos años el desarrollo de aplicaciones móviles ha crecido bastante debido al aumento del uso de smartphones y plataformas como Google Play. Sin embargo, también indica que muchas de estas aplicaciones presentan problemas de calidad, especialmente en la mantenibilidad.
En el estado del arte se revisan varios trabajos previos que analizan la mantenibilidad del software utilizando la norma ISO/IEC 25010. Por ejemplo, se menciona el trabajo de Irrazábal (2012), quien desarrolló un modelo llamado KEMIS para evaluar la calidad del software usando criterios como analizabilidad, estabilidad, cambiabilidad y testeabilidad.
También se menciona a Albeladi y Abdalkareem (2014), quienes evaluaron la mantenibilidad de software usando herramientas como Logiscope, demostrando que es posible medir correctamente esta característica mediante métricas específicas.
Otro estudio importante es el de Malavolta y Verdecchia (2018), quienes analizaron aplicaciones Android y encontraron que los problemas de mantenibilidad aumentan con el tiempo si no se aplican buenas prácticas de desarrollo.
Además, se mencionan investigaciones relacionadas con herramientas para medir la calidad del software, como MetricsReloaded, y estudios que analizan el impacto de patrones de diseño en la mantenibilidad.
Finalmente, el autor también revisa estudios sobre arquitectura limpia, donde se demuestra que su uso mejora el desarrollo, la organización del código y la facilidad de mantenimiento en aplicaciones móviles.
En general, el estado del arte muestra que existen múltiples enfoques para mejorar la mantenibilidad, pero aún hay problemas en aplicaciones reales, lo que justifica la investigación.

## Motivación del autor (críticas del autor a otros trabajos)(0.5 página)

El autor identifica que muchas aplicaciones Android tienen baja calidad debido a una mala arquitectura y a la constante necesidad de actualizaciones. Esto genera problemas como alta complejidad del código, bajo acoplamiento adecuado y dificultad para realizar cambios sin afectar el sistema.
También critica que, aunque existen herramientas y modelos para medir la mantenibilidad, no siempre se aplican correctamente en el desarrollo real de software.
Además, menciona que aplicaciones como CRM Distribución presentaron fallos durante su mantenimiento, lo que causó problemas en los usuarios y retrasos en las correcciones.
Por eso, el autor busca demostrar que aplicar arquitectura limpia junto con la norma ISO/IEC 25010 puede mejorar significativamente la mantenibilidad del software.

## Descripción del aporte del autor(2 páginas)

El principal aporte del autor es demostrar que el uso de arquitectura limpia junto con la norma ISO/IEC 25010 mejora la mantenibilidad de aplicaciones Android.
Para ello, se desarrolló una aplicación llamada Educar Teacher, la cual se comparó con otra aplicación llamada CRM Distribución.
El problema principal que aborda el autor es la baja mantenibilidad en aplicaciones móviles, lo cual afecta directamente la calidad del software, generando errores, dificultad para actualizar y problemas para agregar nuevas funcionalidades.
El impacto de este problema es tanto técnico como organizacional, ya que afecta la eficiencia del desarrollo, aumenta costos y genera insatisfacción en los usuarios.
El autor propone usar arquitectura limpia, que separa el sistema en capas (presentación, dominio y datos), permitiendo un mejor control del código y facilitando su mantenimiento.
Además, utiliza la norma ISO/IEC 25010 para evaluar la calidad del software, específicamente la mantenibilidad, mediante criterios como:
Analizabilidad
Estabilidad
Testeabilidad
Cambiabilidad
El aporte principal es que demuestra con datos reales que estas prácticas sí mejoran la mantenibilidad del software, especialmente en estabilidad.

## Proceso para obtener el aporte que considera el autor(1. página)

El autor utiliza un enfoque cuasi experimental, comparando dos aplicaciones:
Grupo experimental: Educar Teacher
Grupo control: CRM Distribución
Se aplican métricas de mantenibilidad usando herramientas como MetricsReloaded y Logiscope.
También se utilizan fórmulas basadas en la ISO/IEC 25010 para calcular los criterios de mantenibilidad.
Las variables del estudio son:
Variable independiente: arquitectura limpia + ISO/IEC 25010
Variable dependiente: mantenibilidad
Se analizan elementos como:
Clases
Paquetes
Líneas de código
Con esto se obtienen resultados cuantitativos que permiten comparar ambas aplicaciones.

## Proceso para resolver el problema considerado por el autor(1. página)

El autor desarrolla la aplicación Educar Teacher utilizando arquitectura limpia, organizando el sistema en capas:
Capa de presentación
Capa de dominio
Capa de datos
Además, implementa principios SOLID para mejorar la calidad del código.
Luego, evalúa ambas aplicaciones usando métricas específicas como:
Complejidad (WMC)
Acoplamiento
Tamaño de clases
Número de funciones
Finalmente, se calculan los criterios de mantenibilidad y se comparan los resultados entre ambas aplicaciones.

## Métricas que el autor usa y resultado que obtiene. Comentar (los resultados son mejores respecto a otros)(0.5 página)

Los resultados muestran mejoras en la mantenibilidad:
Analizabilidad: mejora de 7%
Estabilidad: mejora de 56%
Testeabilidad: mejora de 0.7%
Cambiabilidad: mejora de 0.9%
Se observa que la mayor mejora se da en la estabilidad, lo que indica que el sistema es más resistente a cambios.
En general, los resultados del aplicativo con arquitectura limpia son mejores que los del aplicativo tradicional
Las métricas utilizadas son adecuadas porque están basadas en estándares internacionales como ISO/IEC 25010.
Sin embargo, se podría mejorar incluyendo más métricas relacionadas con rendimiento o experiencia de usuario.
También se podría evaluar más aplicaciones para obtener resultados más generales.

## Observaciones y/o críticas suyas al artículo(0.5 página)

Una crítica es que el estudio solo compara dos aplicaciones, lo que puede limitar la generalización de los resultados.
Además, algunas métricas están fuera del rango aceptable, lo que indica que aún hay problemas en el código.
También se podría haber incluido pruebas en tiempo real con usuarios.
Como aporte, se podría integrar inteligencia artificial para analizar automáticamente la calidad del código y predecir problemas de mantenibilidad.
También se podría aplicar este modelo en sistemas más grandes o en otras plataformas como web o sistemas empresariales.
Finalmente, se podría combinar arquitectura limpia con metodologías ágiles para mejorar aún más el desarrollo.
